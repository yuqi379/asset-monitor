# 资产测绘监控系统（Hunter + GitHub）

> **文档版本**：v2.1（精简版）
> **更新时间**：2026-05-06
> **实现状态**：Hunter 已完整实现，GitHub 已完整实现

---

## 一、项目概述

### 1.1 目标
搭建多平台网络资产测绘监控系统，实现：
- **日常增量监测**：每日自动发现新增资产，推送钉钉告警
- **月度全量汇总**：每月生成资产报告，覆盖所有监控目标
- **双平台协同**：Hunter（主力）+ GitHub（补充）

### 1.2 核心能力
| 能力 | 说明 |
|------|------|
| 多平台轮询 | Hunter × 10 + ZoomEye × 10 + GitHub × 5 账号并行 |
| 智能去重 | IP + 端口 + 域名 三维度去重 |
| 增量发现 | 仅对新增资产推送告警 |
| 关键词导入 | Excel 三字段（关键词/项目单位/域名）独立查询 |
| 定时调度 | 每日增量 + 每月全量 |

## 二、双平台账号配置

| 平台 | 账号数 | 额度 | 重置周期 | 认证方式 | 角色 |
|------|--------|------|---------|---------|------|
| **Hunter** | ≥1（可配置） | 500积分/账号/天 | 每日零点 | API Key | 主力 |
| **GitHub** | 3-5（可配置） | 5000次/小时/账号 | **每小时重置** | Personal Access Token | 补充维度 |

**账号数量不写死，全部通过 `config.yaml` 配置，可随时增减。**

## 三、双平台API消耗机制

### 3.1 Hunter（鹰图）

```
消耗规则：按"获取到的数据条数"扣积分，不是按查询次数
├── 查询有结果 + 获取1条 → 扣1分
├── 查询有结果 + 获取80条 → 扣80分
└── 查询无结果（total=0）→ 扣1分

触发场景举例：
- 条件A返回80条 → 将80条全部获取 → 扣80分
- 条件B返回200条 → 将200条全部获取 → 扣200分

结论：额度消耗速度取决于返回结果数量
```

**分页分工原理**：
- Hunter API `page` 参数最大 100 条/页
- 单账号每天可获取 500 条（5页 × 100条）
- 多账号分页分工：账号1→page 1~5，账号2→page 6~10 ...
- **前提**：Hunter API 排序固定，换账号查同样的条件不会漏数据

### 3.2 GitHub

```
消耗规则：按API调用次数，不是按结果条数
├── 未认证：60次/小时
├── 已认证（Personal Access Token）：5000次/小时
└── 搜索API（/search/*）：共享速率限制，5000次/小时

主要搜索接口：
├── Code搜索：/search/code?q=关键词
├── Repo搜索：/search/repositories?q=关键词
└── Issue搜索：/search/issues?q=关键词

注：GitHub是补充维度，当Hunter额度不足时使用
```

## 四、账号轮询策略

### 4.1 Hunter
```
├── 每次查询前随机选取一个可用账号
├── 查询过程中额度耗尽（积分不足1条）→ 本批次查询终止，自动切换下一个账号
├── 当天额度用完的账号 → 排除在当日随机池外，直到次日零点重置
└── 所有账号额度耗尽 → 等待次日重置后再继续
```

### 4.2 GitHub
```
├── 每次查询前随机选取一个可用Token
├── 触发速率限制（429）→ 等待至下一小时窗口，自动切换下一个Token
├── 小时内额度耗尽 → 切换下一Token
└── 所有Token耗尽 → 等待下一小时窗口重置
```

## 五、关键词导入逻辑（核心）

### 5.1 Excel 格式

| 列 | 字段名 | 说明 | 示例 |
|----|--------|------|------|
| A | 关键词 | 选填，title/body组合查询 | 电力交易系统 |
| B | 项目单位 | 选填，ICP备案单位 | 某省电力公司 |
| C | 域名 | 选填，domain精确查询 | example.com |
| D | 任务名称 | 选填，默认用关键词/单位/域名 | A公司资产监测 |
| E | 启用状态 | 选填，0=禁用，1=启用，默认1 | 1 |

### 5.2 查询拆分规则

**每行根据非空字段拆成1~3条独立查询，所有查询共享同一个 task_name。**

| Excel 内容 | 拆分为 |
|-----------|--------|
| 关键词=A，项目单位=B，域名=C | 查询1: `title="A" \|\| body="A"` <br> 查询2: `icp.name="B"` <br> 查询3: `domain="C"` |

### 5.3 查询语法对照

| 字段类型 | Hunter 语法格式 | 说明 |
|---------|----------------|------|
| 关键词 | `title="关键词" \|\| body="关键词"` | 只用这两个字段，OR连接 |
| 项目单位 | `icp.name="单位名称"` | ICP备案单位名称 |
| 域名 | `domain="域名"` | 精确域名匹配 |

## 六、去重判断逻辑

```
判断重复的唯一依据：
① IP + 端口 + 域名 三者都相同 → 重复
② 无IP+端口时 → 域名相同 → 重复

去重流程：
同一批次查询结果
    ↓
逐条检查：IP+Port+Domain 是否已存在 SQLite
    ├── 已存在 → 跳过，不告警
    └── 不存在 → 写入 assets 表，推送钉钉告警
```

**matched_fields 推断**：
- 每条资产记录自动标注通过哪些字段命中（title / body / icp.name / domain）
- 用于告警详情展示，方便溯源

## 七、钉钉告警格式

```
同一批次（同一搜索条件）的所有新资产 → 合并为1条推送

### 🚨 新增资产告警

**项目**：A公司-关键词
**平台**：Hunter
**时间**：2025-04-25 14:30
**查询条件**：title="A公司" || body="A公司"
**发现资产（3条）**：

| # | IP:端口 | 域名 | 类型 | 命中字段 |
|---|---------|------|------|---------|
| 1 | 1.2.3.4:443 | api.a.com | SSL证书 | title |
| 2 | 5.6.7.8:80 | www.a.com | Web标题 | body |
| 3 | 9.10.11.12:8080 | oa.a.com | 业务系统 | icp.name |

---
*由 Hermès 资产监测系统推送*
```

## 九、配置文件结构（config.yaml）

```yaml
# ========== Hunter 账号配置 ==========
hunter:
  api_keys:
    - "key_001"
    - "key_002"
    # 有多少写多少，自动随机轮询

# ========== GitHub 账号配置 ==========
github:
  tokens:
    - "ghp_xxxxxxx"
    - "github_pat_xxxxx"
    # 有多少写多少，Token额度为5000次/小时/账号

# ========== 告警配置 ==========
dingtalk:
  webhook: "https://oapi.dingtalk.com/robot/send?access_token=xxx"
  secret: "SECxxxxxxxxxxxxx"

# ========== 查询任务配置 ==========
queries:
  # 关键词查询（title || body 组合）
  - name: "A公司-关键词"
    keyword: 'title="A公司" || body="A公司"'
    enabled: true
  # ICP备案单位查询
  - name: "A公司-ICP备案"
    keyword: 'icp.name="A公司"'
    enabled: true
  # 域名查询
  - name: "A公司-域名"
    keyword: 'domain="a.com"'
    enabled: true
  # GitHub 查询（独立关键词列表）
  - name: "A公司-GitHub代码"
    keyword: '"company.com" language:Python'
    platform: github
    enabled: true

# ========== 调度配置 ==========
scheduler:
  cron_daily: "0 2 * * *"
  cron_monthly: "0 3 1 * *"
  page_size: 100
  delay_between_requests: 2
  github_backup_only: true  # GitHub仅作为额度不足时的补充

# ========== Excel 关键词文件路径 ==========
excel:
  keywords_file: "keywords.xlsx"  # 留空则使用 config.yaml 中的 queries
```

## 十一、SQLite数据库设计

```sql
-- 资产表
CREATE TABLE IF NOT EXISTS assets (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    ip TEXT,
    port INTEGER,
    domain TEXT,
    web_title TEXT,
    protocol TEXT,
    url TEXT,
    province TEXT,
    city TEXT,
    company TEXT,
    platform TEXT,
    found_date TEXT,
    last_seen TEXT,
    source_keyword TEXT,
    source_page INTEGER,
    status TEXT DEFAULT '未检测',
    matched_fields TEXT DEFAULT '',
    UNIQUE(ip, port, domain)
);

-- 已推送记录（去重）
CREATE TABLE IF NOT EXISTS alerted (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    ip TEXT,
    port INTEGER,
    alert_date TEXT,
    platform TEXT,
    source_keyword TEXT
);

-- 项目配置表
CREATE TABLE IF NOT EXISTS projects (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT,
    query_keyword TEXT,
    pages_per_account INTEGER DEFAULT 5,
    enabled INTEGER DEFAULT 1,
    last_run TEXT
);

-- Hunter账号状态表
CREATE TABLE IF NOT EXISTS hunter_keys (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    api_key TEXT,
    idx INTEGER,
    start_page INTEGER,
    end_page INTEGER,
    daily_quota INTEGER DEFAULT 500,
    used_today INTEGER DEFAULT 0,
    last_used TEXT,
    enabled INTEGER DEFAULT 1
);
```

**matched_fields 字段说明**：
- 记录资产通过哪些搜索字段被命中
- 格式：`title,body` 或 `icp.name` 或 `domain`
- 用于告警详情中展示推理过程

## 十二、关键设计决策

| # | 决策 | 说明 |
|---|------|------|
| 1 | **API Key直连调用** | Hunter/GitHub的API只验证Key，无需代理IP |
| 2 | **账号数可配置** | 全部通过config.yaml配置，不写死 |
| 3 | **GitHub作为补充维度** | 当Hunter额度不足时使用GitHub代码搜索作为补充 |
| 4 | **去重逻辑** | IP+Port+Domain 三维度去重 |
| 5 | **随机延时** | 每次请求间隔2秒，模拟真人操作 |
| 6 | **Hunter按条扣分** | 每获取一条数据扣1分 |
| 7 | **Excel三字段独立查询** | 关键词/项目单位/域名各自独立，互不影响 |
| 8 | **matched_fields追溯** | 每条资产记录标注命中的搜索字段 |

## 十三、运行方式

### 13.1 开发模式（前后端分离）
```bash
# 终端1：启动后端
cd asset_monitor
python api_server.py

# 终端2：启动前端
cd frontend
npm run dev
# 访问 http://localhost:5173
```

### 13.2 生产模式（一进程搞定）
```bash
# 前端构建
cd frontend && npm run build

# 只需启动后端，Vue 前端由 Flask 直接服务
cd asset_monitor
python api_server.py
# 访问 http://localhost:5000
```

### 13.3 资产导出
```bash
# 命令行导出
python export_assets.py --output report.xlsx

# API导出
curl http://localhost:5000/api/export/excel -o assets.xlsx
```

### 13.4 定时任务（crontab）
```bash
# 每日凌晨2点增量监测
0 2 * * * cd /path/to/asset_monitor && python main.py >> logs/daily.log 2>&1

# 每月1号凌晨3点全量报告
0 3 1 * * cd /path/to/asset_monitor && python export_assets.py --output reports/$(date +\%Y-\%m).xlsx >> logs/monthly.log 2>&1
```
